**Version:** 0.1.0  
**Date:** 2026-04-05  
**Purpose:** Publication-grade rationale for all design decisions in this

## 1. Project Overview

This project benchmarks waveform features for the **input layer** of a
real-time spiking neural network (SNN) spike discriminator. The scientific
question is: *which waveform features best discriminate individual neurons and
reject noise under strict computational constraints?*

The benchmark is conducted on:
1. **Synthetic waveforms** generated by LFPy/NEURON biophysical forward models
   (Hines & Carnevale 1997; Hagen et al. 2018) — controlled ground truth
2. **Real sorted units** from human intracranial microwire recordings sorted
   with osort (Rutishauser et al. 2006) with hand curation and noise labeling

---

## 2. Why an Input Layer, Not a Full SNN

The full SNN architecture includes attention neurons and noise-filter neurons
(see `docs/spike_feature_validation.yaml §snn_mapping_goal`). However, the
**input layer is the computational bottleneck**: it must execute within ~1 ms
per threshold crossing at ≥ 30 kHz sampling. The downstream layers operate on
population codes that are already highly compressed.

**Decision:** Benchmark only the input layer feature extraction. This isolates
the compute-vs-discrimination tradeoff and allows us to identify the minimal
sufficient feature set before designing the full network.

## 3. Feature Selection Rationale

### 3.1 Physics of Extracellular Recording

An extracellular microwire at distance *r* from a neuron records a signal
proportional to the transmembrane current distribution integrated with a
distance-dependent weighting (line-source approximation; Holt & Koch 1999):

$$V_{ext}(\mathbf{r}, t) = \frac{1}{4\pi\sigma} \sum_{n} \frac{I_n(t)}{r_n}$$

The resulting waveform is a biphasic or triphasic signal whose shape encodes:

| Physical quantity | Feature | Tier |
|---|---|---|
| Amplitude of currents | `amplitude` | 1 |
| Rate of current change | `first_derivative` | 1 |
| Turning point location | `second_derivative` | 1 |
| Total signal energy | `short_window_energy` | 2 |
| Instantaneous frequency | `teager_energy` | 2 |
| Spike duration | `half_width`, `full_width` | 3 |
| Waveform asymmetry | `biphasic_ratio` | 3 |

### 3.2 Feature Tiers (from `docs/features.yaml`)

**Tier 1 — Ultra-fast streaming** (≤ 3 ops/sample, causal):  
`amplitude`, `first_derivative`, `second_derivative`

These correspond to the waveform's *position*, *velocity*, and *acceleration*
— the three quantities that fully characterize waveform shape dynamics at any
instant. They are the minimal viable set.

**Tier 2 — Fast streaming** (≤ 6 ops/sample, causal, windowed):  
`absolute_window_sum`, `short_window_energy`, `trough_amplitude`, `peak_amplitude`, `max_rising_slope`, `max_falling_slope`

**Tier 3 — Event-based** (require complete snippet, higher latency):  
`trough_to_peak_time`, `half_width`, `biphasic_ratio`, `baseline_rms`, `signed_area`

### 3.3 Why Not PCA/Wavelets as Primary Features?

PCA and wavelet decompositions are excellent offline features but have
significant real-time drawbacks:
- **PCA** requires a covariance matrix from a calibration set and a matrix
  multiply per snippet
- **Wavelets** require O(T log T) per snippet and have poor causality properties
- Both hide the physical interpretation: it is unclear which waveform property
  they measure

**Decision:** PCA/wavelets are included as Tier 4 optional comparisons to
establish an upper bound on achievable performance, not as primary candidates.

## 4. Input Layer Architecture: Population Code

### 4.1 Why Population Code (Not Rate Code)

Rate coding requires multiple spikes to transmit a scalar value — incompatible
with single-shot discrimination. Latency coding (earlier spike = stronger
signal) is fast but fragile to noise. Population coding is the most robust
approach for encoding analog feature values as binary SNN activity.

Each scalar feature value is encoded by *N_bins* input neurons whose preferred
values span the feature range. Neuron *b* fires if the feature value *x* falls
within its tuning width of center *c_b*:

$$a_b = \exp\left(-\frac{(x - c_b)^2}{2\sigma_b^2}\right)$$

### 4.2 Quantile Bin Initialization (vs. Uniform or K-means)

**Decision: quantiles from calibration spikes**

Rationale:
- **Uniform spacing** would waste neurons on under-populated feature ranges
  (e.g., extreme amplitude values rare in practice)
- **K-means** would better capture multimodal distributions but requires an
  iterative algorithm and is sensitive to initialization
- **Quantile spacing** guarantees equal neuron load across the data
  distribution, maximizes information per neuron under the assumption that the
  calibration set is representative, and has O(N log N) cost with a single
  sort

The quantile levels are evenly spaced in [0, 1]: `np.linspace(0, 1, N_bins)`.
Bin widths are set to half the distance to the adjacent bin center, scaled by
`sigma_scale` (default 1.0). This ensures smooth coverage with minimal
overlap.

**Hyperparameter: `n_bins = 10`** — sufficient resolution for continuous
scalar features; 10 neurons × 6 features = 60 input neurons total for Tier 2.
Increasing to 20 improves discrimination marginally but doubles input size.

**Hyperparameter: `sigma_scale = 1.0`** — corresponds to Gaussian width equal
to half the inter-quantile spacing. Values > 1 produce smoother, more
tolerant tuning (useful when calibration set is small). Values < 1 produce
sharper gates (risk of gaps between bins if data distribution shifts).

### 4.3 Discriminant Score

The WeightBank computes a weighted average of per-feature max activations:

$$\text{score} = \frac{\sum_f w_f \cdot \max_b a_{f,b}}{\sum_f w_f}$$

A real spike produces high activation in most feature bins (≥ 0.5 score).
Noise produces low, inconsistent activation. The threshold (default 0.5) is
tunable and should be set from ROC analysis on a validation set.

## 5. Why Numba (Not Cython, PyPy, or C Extensions)

| Option | Pros | Cons |
|---|---|---|
| **Numba** | Pure Python source, JIT, parallel, LLVM | JIT warmup overhead (~1 s first call) |
| Cython | Fast, no JIT overhead | Requires separate compilation step |
| C extension | Maximum control | High maintenance burden |
| PyPy | Drop-in | Poor numpy interop, complex deployment |
| Vectorized NumPy | Simple | No loop-level parallelism, hidden copies |

**Decision: Numba** because:
1. Source code remains in Python (readable for publication)
2. `@numba.njit(parallel=True, fastmath=True)` generates LLVM IR that matches
   hand-written C in throughput for simple inner loops
3. `numba.prange` parallelizes across snippets with zero code change
4. The JIT warmup is amortized across the full benchmark run

All kernels use `cache=True` to persist compiled artifacts across runs.

## 6. Zero-Copy Memory View Strategy

All feature kernels accept:
- `waveforms: float32[N, T]` — input, C-contiguous, NOT copied
- `out: float32[N, T]` — pre-allocated output buffer

The caller allocates output buffers once before the benchmarking loop and
reuses them. No `np.empty()` calls occur inside JIT functions. This pattern:
- Eliminates GC pressure on repeated calls
- Keeps all data in L2/L3 cache between Tier 1 and Tier 2 computations
- Is directly portable to C/FPGA where fixed-size buffers are mandatory

**Exception:** scalar summary extraction from time-series features uses a
single pass that directly computes min/max without materializing the full
derivative array. This is implemented in `batch_extract_scalar_features`.

## 7. Data Storage Decisions

| Data type | Format | Rationale |
|---|---|---|
| Raw waveforms | `.npz` (float32) | Compact, numpy-native, no schema required |
| Feature tables | `.parquet` | Columnar, inspectable by pandas/polars/DuckDB |
| Labels & metadata | `.parquet` | Same row-aligned format as feature tables |
| Configs | `.yaml` | Human-readable, version-control friendly |
| Profiling results | `.json` | Lightweight, key-value structure |
| WeightBank params | `.json` | Human-readable bin centers/widths |

**All results are stored to `data/`** with timestamped subdirectories. This
means every benchmark run is fully reproducible and independently inspectable
without running any code.

## 8. Why Skip Attention and Noise Filter Layers

The full SNN design includes:
- **Input layer** (this project) — feature-to-spike population encoding
- **Attention layer** — gate on spike identity confidence
- **Noise filter layer** — learned veto for artifact patterns

For initial benchmarking we skip downstream layers because:
1. Input layer performance determines the ceiling on downstream discrimination
2. Adding attention before validating input features hides poor features
3. Noise rejection in the input layer (via dedicated noise-class bins) is
   sufficient for initial validation

## 9. Hyperparameter Documentation

All hyperparameters are in `configs/default.yaml`. Key decisions:

| Parameter | Value | Rationale |
|---|---|---|
| `sampling_rate_hz` | 30000 | Standard Utah/microwire rate; 20 kHz also valid |
| `snippet_length_samples` | 64 | ~2.1 ms at 30 kHz; captures full spike waveform |
| `pre_peak_samples` | 20 | 0.67 ms pre-trough; captures onset and rising slope |
| `post_peak_samples` | 44 | 1.47 ms post-trough; captures repolarization and AHP |
| `threshold_multiplier` | 4.5 | 4–5× RMS is standard in extracellular sorting literature |
| `baseline_window_samples` | 512 | ~17 ms rolling window for RMS estimation |
| `window_size_samples` | 16 | 0.53 ms for windowed features (AWS, energy) |
| `n_bins` | 10 | Population-code resolution; 10 neurons per feature |
| `sigma_scale` | 1.0 | Gaussian tuning width = half inter-quantile spacing |
| `discriminant_threshold` | 0.5 | Initial default; tune from ROC on validation set |

## 10. Validation Strategy

### Phase 1 (this implementation)
- Synthetic LFPy waveforms with known labels
- Tier 1–2 feature benchmarking
- WeightBank calibration and scoring
- Op count + timing profiling

### Phase 2 (next)
- Load hand-curated osort units (`.mat` format)
- Validate WeightBank trained on synthetic against real units
- Noise rejection using osort-rejected events
- Update feature rankings with real-data performance

### Phase 3
- Session generalization (train on one recording, test on another)
- Unit identity generalization (leave-one-unit-out)
- Embed top features into SNN input layer config

## 11. Compute Cost Analysis

Static operation counts per sample (from `docs/features.yaml`):

| Feature | Adds | Muls | Mem reads | Total ops |
|---|---|---|---|---|
| amplitude | 0 | 0 | 1 | 1 |
| first_derivative | 1 | 0 | 2 | 3 |
| second_derivative | 2 | 1 | 3 | 6 |
| absolute_window_sum | 1 | 0 | 1 | 2 |
| short_window_energy | 1 | 1 | 1 | 3 |
| teager_energy | 1 | 2 | 3 | 6 |

The minimal Tier 1 set (amplitude + d1 + d2) requires **10 operations per
sample** = 640 ops for a 64-sample snippet. At 30 kHz with 100 threshold
crossings per second, this is **64,000 ops/second** — trivially within any
processor budget.

## 12. LFPy Synthetic Generation

LFPy simulates extracellular potentials using the line-source approximation:

$$V_{ext}(P, t) = \frac{1}{4\pi\sigma} \sum_n \int \frac{I_n(t)}{|\mathbf{r} - \mathbf{r}_n|} dl$$

The ball-and-stick Hodgkin-Huxley model used here consists of:
- **Soma**: 20 µm diameter sphere, HH channels (gNa=0.12, gK=0.036 S/cm²)
- **Dendrite**: 200 µm × 3 µm, passive leak (g_pas=5×10⁻⁴ S/cm²)

The electrode is placed at a grid of distances (10–150 µm) and angles (0–360°)
around the soma. This produces waveforms that vary in:
- **Amplitude** (inversely with distance²)
- **Biphasic shape** (depends on angle relative to soma/dendrite axis)
- **Trough-to-peak time** (relatively stable — good unit identity feature)

Noise is added as bandlimited Gaussian noise at levels 0, 1, 3, 5, 10 µV RMS.
